Overview
02 — Concept
What is Blumetara.ai?
An AI-driven mobile wellness co-pilot designed to translate complex laboratory report values into understandable wellness checklists, routines, and ongoing coaching loops.
Laboratory Translation
Converts unstructured PDFs and report images into clear structured items showing reference range anomalies.
Intelligent Chat coach (TARA)
Retrieves the user's specific health goals, laboratory background, and active reminders to answer queries safely.
Reminders & Adherence
Presents drink water reminders, medicine notification triggers, daily reflection prompts, and workout schedules synced with diagnostic results.
Core Value Proposition
By connecting AWS Textract OCR, MongoDB RAG semantic search, and the Gemini API, Blumetara bridges the gap between getting diagnostic results and executing a daily healthy routine.

Model 2 Workflow
13 — Event Driven Pipelines
What Changes Compared to Model 1?

In Model 1, the backend may directly trigger background tasks. In Model 2, the backend publishes events, triggering a robust, multi-stage processing pipeline.

Durable Event list
ReportUploaded • OCRCompleted • EmbeddingCompleted • ReminderDue • NotificationRequested • PaymentWebhookReceived
1
Upload & Save
User uploads report. FastAPI immediately saves metadata and publishes ReportUploaded event.
2
OCR Parsing
OCR worker consumes event, runs extraction, and publishes OCRCompleted event.
3
Chunking & Summaries
Embedding worker writes vectors; summary worker generates report metrics card, and user is notified.

14 — Future proofing
Why Model 2 is Better for Expansion
Because future features can plug directly into existing queues and events without refactoring the core APIs:
Scenario A: Weekly insights

When adding weekly health insights:

1
GoalProgressUpdated Event
New Insights worker listens to the stream, generates a weekly gap analysis, and adds it to the user's dashboard. Core goal code remains untouched.
Scenario B: Caregiver family Alerts

When adding family caregiver features:

1
CriticalReminderMissed Event
A new Caregiver alert worker listens to missed medicine triggers, immediately sending push messages to family members.

Decision Backlog
26 — Engineering Choices
Architecture Decisions Pending Confirmation
1. Compute & Cache
• FastAPI Backend: AWS App Runner (for simplicity) vs ECS Fargate (for strict control).

• Redis Cache: Docker Redis for local environments; AWS ElastiCache for production.
2. Async & Database
• Async Queue: Celery/Redis for simple monolith vs SQS/EventBridge for hybrid async.

• Vector DB: MongoDB Atlas Vector Search for MVP to keep stack simpler.
3. AI, OCR & Cloud
• AI Providers: Abstract model calls behind an adapter layer to support Gemini/OpenAI.

• OCR Engine: Google Vision for basic uploads vs AWS Textract/Doc AI.

• Cloud Strategy: Start AWS-first.

Staged Roadmap
27 — Recommendation
Staged Engineering Roadmap
1. Launch Build
Model 1: Modular Monolith

Fastest and most cost-efficient way to launch MVP + V1 capabilities while keeping strict codebase domain modularity on AWS.

2. Mid-Term Scale
Model 2: Hybrid Async

Decouple heavy text OCR extraction and vector embedding generation into durable SQS workers as file volume scales.

3. Enterprise Expansion
Model 3: Vertex AI Platform

Transition to GCP Vertex AI (Gemini, Document AI, Vertex Search) for clinical-grade OCR after closing seed/investor rounds.

Core Engineering Principle
A modular monolith is not a throwaway prototype. By separating domains cleanly on day one, we can extract modules into microservices later only when load or team size demands it.
Meeting Script

As active user scale increases, we have a clean, documented upgrade pathway to move heavy tasks to durable SQS queues (Model 2) and eventually transition into an enterprise-grade Google Vertex AI platform (Model 3) once we secure funding."
